# 범용 IT/이공계 직무 가이드

금융 IT가 아닌 일반 IT/이공계 직무 지원 시 참고하는 보강 가이드.

## 목차

1. 기본 관점
2. 지원동기 작성 포인트
3. 입사 후 포부 작성 포인트
4. 직무 역량 문항 포인트
5. 자주 하는 실수

---

## 1. 기본 관점

IT/이공계 자소서는 기술만 설명하는 문서가 아니다.
아래 4가지를 함께 보여줘야 한다.

- **지식**: 무엇을 배웠는가
- **기술**: 실제로 어떻게 구현·분석·검증했는가
- **태도**: 어떤 판단과 마인드로 임했는가
- **사업/고객 이해**: 이 기술이 누구에게 어떤 가치를 주는가

---

## 2. 지원동기 작성 포인트

### 기본 구조

1. 회사가 집중하는 제품/사업/기술 방향 한 줄 요약
2. 그 방향이 왜 중요하다고 보는지 내 해석
3. 관련 경험 또는 준비 과정
4. 해당 직무에서 보탤 수 있는 기여

### 좋은 연결 방식

- **기술 관심형**: "○○ 기술을 공부하며, 귀사가 그 기술을 실제 서비스/제품에 연결하는 방식에 관심을 가졌습니다"
- **문제 해결형**: "귀사가 풀고 있는 ○○ 문제는 기술적으로도 난도가 높고 사용자 가치도 크다고 판단했습니다"
- **제품/고객 가치형**: "이 기술이 고객 경험, 안정성, 운영 효율을 어떻게 바꾸는지에 끌렸습니다"

### 직무별 한 줄 포인트

- **백엔드/플랫폼**: 안정성, 확장성, 성능, 운영 효율
- **프론트엔드**: 사용자 경험, 상태 관리, 협업, 전달 품질
- **AI/데이터**: 문제 정의, 데이터 품질, 실험 설계, 서비스 적용
- **QA/품질**: 재현, 검증 체계, 리스크 선제 차단, 품질 기준
- **인프라/DevOps/SRE**: 자동화, 관측 가능성, 장애 대응, 배포 안정성

## 3. 입사 후 포부 작성 포인트

입사 후 포부는 WHAT/HOW/WHY로 쓴다.

- **WHAT**: 어떤 과업에서 강점을 만들 것인가
- **HOW**: 어떤 역량과 방식으로 수행할 것인가
- **WHY**: 그것이 회사와 사용자에게 왜 중요한가

### 좋은 포부 예시의 공통점

- 기술 이름만 나열하지 않는다
- 실제 업무 범위 안에서 말한다
- 고객 가치나 운영 가치로 마무리한다

### 나쁜 포부 패턴

- "최고의 개발자가 되겠습니다"
- "모든 기술을 빠르게 습득하겠습니다"
- "혁신적인 서비스를 만들겠습니다"

위 문장들은 목표는 있으나 직무 범위와 실행 방법이 비어 있다.

---

## 4. 직무 역량 문항 포인트

기술 역량 문항에서는 아래 5개를 확인한다.

1. 무엇을 구현/분석/설계했는가
2. 왜 그 방법을 택했는가
3. 어떤 어려움이 있었는가
4. 어떻게 검증했는가
5. 결과가 사용자/팀/서비스에 어떤 의미였는가

### 권장 문장 흐름

- 문제 상황
- 원인 파악
- 설계/구현/검증 방식
- 결과
- 실무적 인사이트

---

## 5. 자주 하는 실수

- 기술 스택만 나열하고 실제 판단이 없다
- "협업했다"만 있고 어떤 기준으로 합의했는지 없다
- 성능 수치는 있는데 사용자 가치가 없다
- 지식은 많은데 태도와 마인드가 안 보인다
- 회사 기술 방향과 내 경험이 연결되지 않는다
