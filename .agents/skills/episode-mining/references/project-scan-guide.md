## 파일 탐색 순서

1. **프로젝트 루트** 먼저 스캔 → README, 설정 파일 등 전체 윤곽 파악
2. **문서 폴더** (docs/, reports/) → 보고서, 발표 자료, 설계 문서, 이슈 분석
3. **의존성 파일** → 기술 스택 파악
4. **소스 코드 구조** → 폴더명/파일명으로 아키텍처 윤곽 파악
5. **인프라 파일** → Dockerfile, CI/CD, 웹서버 설정
6. **버전 관리** (.git) → 커밋 로그, PR 내역

## 문서 우선 + 코드 보완 원칙

이 스킬의 기본값은 **코드 정독이 아니라 문서 우선 스캔**이다.
문서, README, 보고서, git 로그만으로도 자소서 후보를 빠르게 좁힐 수 있기 때문이다.

### 1차 스캔: 문서·로그 기반 후보 발굴

아래 자료로 먼저 에피소드 후보를 추린다:
- README, 설계 문서, 발표 자료
- 이슈 분석 문서, 회고 문서
- 커밋 로그, PR 제목, 의존성/인프라 설정

이 단계에서는 `무슨 문제를 풀었는지`, `무슨 결과가 있었는지`, `어떤 기술 범주인지`만 빠르게 잡는다.

### 2차 스캔: 코드 기반 후보 검증·보완

코드는 아래 경우에만 제한적으로 읽는다:
- 문서에는 기능만 있고 **왜 그렇게 구현했는지**가 빠져 있을 때
- 커밋에는 수정 흔적이 있는데 **기술적 깊이**가 설명되지 않을 때
- 자소서 가치가 큰 후보 에피소드가 2~3개로 압축된 뒤, **면접 방어 가능한 디테일**을 보강해야 할 때

### 코드 스캔 중단 기준

아래에 해당하면 더 깊게 읽지 않는다:
- 단순 CRUD나 프레임워크 보일러플레이트만 보일 때
- 구현은 보이지만 자소서 문장으로 설명할 차별점이 없을 때
- 사용자가 직접 설명하지 못할 가능성이 큰 저수준 디테일일 때

## 자동 추출 vs 사용자 질문 구분

### ✅ 파일에서 추출 가능한 항목

파일의 텍스트에 직접적으로 드러나는 **사실(fact)** 정보:
- 프로젝트 목표, 기술 스택, 기간
- 겪은 문제 (이슈 트래커, 커밋 로그)
- 해결 시도 (코드 변경 내역)
- 성과 수치 (보고서 결론)

### ❓ 반드시 사용자에게 물어야 하는 항목

파일에 드러나지 않는 **맥락(context)·감정·판단** 정보:
- 팀 구성과 **본인의 구체적 역할** (4)
- 문제를 **어떤 과정으로** 발견했는지 (7)
- 본인의 **기여도** (몇 %인지)
- **감정** — 가장 힘들었던 점, 뿌듯했던 순간
- **배운 점**과 **직무 적용 가능성** (13, 14)

## 코드 2차 탐색과 번역 원칙

문서/README/발표자료로 맥락을 파악한 뒤에는, 코드 2차 탐색으로 차별점의 **실체**를 확인한다.

### 1) 코드 2차 탐색의 목적

- 문서에 적힌 설계가 실제 코드에도 반영됐는지 확인
- 문서/로그에 드러나지 않은 숨은 강점을 보완
- 차별점이 되는 구현 포인트를 찾되, 자소서에는 직접 노출하지 않을 수준까지 정제
- "기술 나열"이 아니라 "설계 판단"을 뽑아내기

### 2) 무엇을 찾는가

| 탐색 대상 | 확인 질문 |
|---|---|
| 상태 전이 | 어디서 어떤 조건으로 상태가 바뀌는가? |
| 트랜잭션 경계 | 커밋 전/후를 어떻게 나누는가? |
| 중복 방지 | 재시도·중복 요청을 어떻게 다루는가? |
| 사용자 흐름 연결 | 계산/처리 결과가 다음 화면·다음 행동과 어떻게 연결되는가? |
| 제약 반영 | 시간, 비용, 일정, 팀 규모, 레거시 제약이 코드에 어떻게 반영됐는가? |
| 근본 원인 추적 | 표면 증상이 아니라 어떤 코드 경로가 병목/오류를 만들었는가? |

### 3) 3층 번역 규칙

코드 2차 탐색 결과는 반드시 아래 3층으로 번역한다.

| 층위 | 질문 | 예시 |
|---|---|---|
| **증거층** | 코드에서 무엇을 확인했는가? | `AFTER_COMMIT`, `cachedMaxLoanAmount`, `FOR UPDATE` |
| **해석층** | 이게 왜 중요한가? | 커밋 경계, 결과 복원, 락 범위까지 고려한 설계 |
| **서술층** | 자소서에는 어떻게 쓸 것인가? | 실행 안정성까지 고려해 처리 흐름을 설계했다 |

### 4) 지엽화 방지 규칙

- 구현명이 없어도 의미가 전달되지 않으면 아직 번역이 덜 된 것이다
- 자소서에는 내부 클래스명/컬럼명/메서드명을 거의 쓰지 않는다
- 기술 포인트 하나당 반드시 **사용자 가치 / 운영 가치 / 업무 가치** 중 하나를 붙인다
- "Redis Streams를 썼다"보다 "긴 작업을 사용자 요청과 분리해 응답성과 실행 안정성을 함께 확보했다"를 우선한다
- "cachedMaxLoanAmount를 저장했다"보다 "정밀 계산 결과가 다음 의사결정 화면에도 일관되게 이어지도록 했다"를 우선한다

### 5) 출력 형식

코드에서 찾은 차별점은 아래처럼 정리한다:

```
🧩 차별 포인트
├── 증거층: [코드/문서에서 확인한 구현 근거]
├── 해석층: [이 근거가 보여주는 설계 판단]
├── 서술층: [자소서용 문장]
└── 면접 메모: [구현 디테일을 설명할 때만 쓸 키워드]
```

## 이공계/IT 추가 질문 축

기술 단서를 찾았더라도, 아래 질문이 빠지면 자소서가 기술 나열로 끝나기 쉽다.

### 반드시 확인할 질문

- **기술 선택 이유**: "왜 이 기술/구조를 택했나요? 다른 대안은 왜 배제했나요?"
- **시장·고객 맥락**: "이 기능이나 개선이 사용자, 고객, 운영자에게 왜 중요했나요?"
- **제약 조건**: "시간, 인력, 비용, 레거시 제약 중 무엇이 가장 컸나요?"
- **판단 기준**: "좋은 해결책인지 어떤 기준으로 판단했나요?"
- **협업 방식**: "기획/디자인/현업/동료와 어떤 기준으로 합의했나요?"
- **자소서 번역**: "이 기술 포인트를 면접 답변이 아니라 자소서 문장으로 바꾸면 어떻게 말할 수 있나요?"
- **지엽화 점검**: "이 표현이 구현 디테일에 너무 갇혀 있지는 않나요?"

### 질문이 특히 중요한 상황

- README에는 기능만 있고, 왜 그렇게 만들었는지 설명이 없을 때
- 커밋에는 수정 흔적이 있는데, 본인 설계 판단이 드러나지 않을 때
- 성능 개선 수치가 있지만 사용자 가치가 연결되지 않을 때
- 프로젝트가 기술적으로는 좋아 보이는데 직무 적합성이 약할 때

## 포트폴리오 보강 가이드

프로젝트 파일 분석 후, 현재 프로젝트에 **없지만 추가하면 자소서 소재가 되는 것**을 제안한다.
사용자가 추가 실험·개발을 통해 에피소드를 보강할 수 있도록 안내한다.

### 보강 항목 체크리스트

| 영역 | 확인 포인트 | 없으면 보강 제안 |
|---|---|---|
| **성능 측정** | 부하 테스트 결과가 있는가? | k6, JMeter, Artillery 등으로 테스트 → Before/After 정량 수치 확보 |
| **모니터링** | 메트릭/로그 수집 구성이 있는가? | Prometheus+Grafana, ELK 등 도입 → "시스템 안정성 확보" 어필 |
| **테스트 코드** | 단위/통합/E2E 테스트가 있는가? | 핵심 로직 테스트 추가 + 커버리지 측정 → "품질 관리 의식" |
| **문서화** | API 문서, 아키텍처 문서가 있는가? | Swagger, 설계 문서 작성 → "커뮤니케이션 역량" |
| **보안** | 인증/인가, 입력 검증이 있는가? | OWASP 대응, SQL Injection 방어 → "보안 의식" |
| **캐싱** | 캐시 레이어가 있는가? | 캐시 도입 후 응답 시간 개선 수치 → "성능 최적화 역량" |
| **DB 최적화** | 인덱스, 쿼리 최적화 흔적이 있는가? | EXPLAIN 분석 → 인덱스 추가 → 쿼리 시간 단축 에피소드 |
| **배포 자동화** | CI/CD 파이프라인이 있는가? | GitHub Actions/Jenkins 구축 → "자동화를 통한 효율화" |
| **에러 처리** | 글로벌 에러 핸들링이 있는가? | 표준 에러 응답 + 로깅 전략 → "안정성 설계" |
| **확장성** | 수평 확장 고려가 있는가? | 로드밸런싱, 세션 공유, 메시지 큐 도입 실험 |

### 보강 제안 출력 형태

```
💡 포트폴리오 보강 제안

현재 프로젝트에서 추가하면 자소서 소재가 되는 항목:

1. ⚡ [영역명] — [현재 상태]
   → [구체적 보강 방법]
   → 자소서 활용: [어떤 문항에 활용 가능한지]
   → 예상 소요: N~M시간
```

> 제안은 **자소서 활용 가능성이 높은 순서**로 정렬한다.
> 각 제안에 **예상 소요 시간**을 명시하여 사용자가 판단할 수 있게 한다.

```
🔧 기술 심화 질문:

12. 시간·비용·기술 제약 중 무엇 때문에 현재 방식을 택했나요?
13. 코드에서 확인된 [설계/예외 처리/최적화 포인트]는 본인이 직접 판단한 것인가요?
```
